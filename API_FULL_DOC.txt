Solana Trust Engine: Полная документация по API

1) Требования и подготовка окружения
- Python 3.10+ установлен в системе
- Установлены зависимости: fastapi, uvicorn, motor, httpx, pydantic, pydantic-settings
- Доступность MongoDB: по умолчанию mongodb://localhost:27017
- База сырых данных: Rawbd, база проверенных данных: Clearbd
- Порты:
  - Внутренний API (pipeline): 8000
  - Доступный API (read-only, с токеном): 9001

2) Запуск доступного API (порт 9001)
- Из корня проекта:
  - uvicorn api_service.apimain:app --app-dir src --host 0.0.0.0 --port 9001
- При старте:
  - Подключение к MongoDB
  - Проверка индексов
  - Готовность к приёму запросов
- Требуется заголовок X-API-Token для всех эндпоинтов

3) Генерация API-токена
- Команда из корня проекта:
  - python .\src\api.py --label <метка>
- Токен сохраняется в коллекцию Clearbd.api_tokens с active=true
- Сгенерированное значение токена выводится в консоль; используйте его в заголовке X-API-Token

4) Эндпоинты доступного API (9001)
- GET /verified
  - Параметры: count (1..100, по умолчанию 10), mode (mainnet|testnet|devnet)
  - Заголовок: X-API-Token (обязателен)
  - Ответ: массив объектов с полями wallet, score, confidence, reason_codes
  - Назначение: получить последние проверенные кошельки и их агрегированный балл
- GET /verified/wallet/{address}
  - Параметры: mode (mainnet|testnet|devnet), wait (0..60, по умолчанию 15)
  - Заголовок: X-API-Token (обязателен)
  - Поведение:
    - Если записи нет в Clearbd, выполняется анализ кошелька с обращением к RPC/Helius
    - Если wait > 0, эндпоинт будет несколько раз проверять появление результата
    - При успехе возвращает wallet, score, confidence, reason_codes
    - При отсутствии релевантных данных может вернуть 404 Not Found

5) Внутренний API (порт 8000)
- Запуск:
  - python .\src\app\api\main.py
  - В интерактивном запуске указывается режим сети (devnet/testnet/mainnet)
- Эндпоинты:
  - GET /wallet/{address} — ставит кошелек в очередь анализа и возвращает статус, если запись ещё отсутствует
  - POST /batch — принимает массив addresses (до 100), ставит все в очередь анализа
  - GET /explain/{address} — возвращает детальные features, scores, confidence (если запись существует)
- Назначение: управление пайплайном анализа; не требует X-API-Token

6) Параметры и режимы
- mode:
  - mainnet — основная сеть Solana
  - testnet — тестовая сеть Solana
  - devnet — разработческая сеть Solana
- wait (для /verified/wallet/{address}):
  - количество секунд на периодическую проверку появления результата
  - фактически это несколько попыток с небольшим интервалом

7) Примеры запросов (порт 9001)
- PowerShell:
  - Invoke-RestMethod -Uri "http://127.0.0.1:9001/verified?count=10&mode=mainnet" -Headers @{ "X-API-Token" = "<ТОКЕН>" }
  - Invoke-RestMethod -Uri "http://127.0.0.1:9001/verified/wallet/<АДРЕС>?wait=15&mode=mainnet" -Headers @{ "X-API-Token" = "<ТОКЕН>" }
- curl:
  - curl -H "X-API-Token: <ТОКЕН>" "http://127.0.0.1:9001/verified?count=10&mode=mainnet"
  - curl -H "X-API-Token: <ТОКЕН>" "http://127.0.0.1:9001/verified/wallet/<АДРЕС>?wait=15&mode=mainnet"

8) Статусы и типичные ответы
- 200 OK:
  - /verified — успешная выдача списка
  - /verified/wallet/{address} — успешная выдача результата после анализа или если запись уже была
- 401 Unauthorized:
  - Отсутствует заголовок X-API-Token или токен неактивен/неверный
- 400 Bad Request:
  - Неверный формат адреса кошелька
- 404 Not Found:
  - Для /verified/wallet/{address} — анализ не дал релевантных транзакций или запись отсутствует после периода ожидания

9) Взаимодействие с блокчейном и внешними сервисами
- Основной источник данных — Solana JSON-RPC (getSignaturesForAddress, getTransaction)
- Дополнительно используется Helius Enhanced API:
  - База: https://api.helius.xyz/v0 (mainnet), https://api-devnet.helius.xyz/v0 (devnet)
  - Эндпоинт: /addresses/{wallet}/transactions с params: api-key, limit
  - При ошибке 4xx (например, 400 Bad Request) сервис логирует текст ответа и продолжает анализ через стандартный RPC
- Обработка rate limit:
  - Реализованы экспоненциальные паузы при 429 и соответствующих RPC-кодах

10) Особенности анализа и фильтрации
- При наличии списка программ (program_ids) пайплайн пытается выделить подписи транзакций, связанные с ними
- Если прямых совпадений нет, выполняется общий анализ без пустого фильтра по сигнатурам, чтобы не блокировать результат
- Малое число транзакций снижает уверенность результата (confidence)

11) Производительность и лимиты
- Пакетная обработка подписей транзакций
- Ограничение глубины рекурсивного анализа связей (поиск funder) до 3
- Небольшие задержки между батчами для снижения нагрузки

12) Безопасность
- Секреты (API-ключ Helius, токены доступа) должны передаваться только через переменные окружения или конфигурацию .env
- Не логируйте токены и ключи в открытый вывод
- Доступный API ограничен заголовком X-API-Token

13) Диагностика и устранение неполадок
- ModuleNotFoundError: при запуске uvicorn — используйте ключ --app-dir src или запускайте из корня проекта
- 400 Bad Request от Helius:
  - Проверьте корректность URL и параметров (api-key, limit)
  - Убедитесь в валидности API-ключа и отсутствии превышения квот
  - Сервис автоматически продолжит через стандартный RPC
- 404 Not Found на /verified/wallet/{address}:
  - Проверьте корректность адреса и режим сети
  - Увеличьте значение wait
  - Возможна ситуация отсутствия релевантных транзакций в сети

14) Быстрый чек-лист запуска
- Установить зависимости
- Запустить MongoDB
- Сгенерировать токен: python .\src\api.py --label <метка>
- Запустить доступный API: uvicorn api_service.apimain:app --app-dir src --host 0.0.0.0 --port 9001
- Выполнить запрос с заголовком X-API-Token
